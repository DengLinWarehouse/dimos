# DimOS 云端设计 PRD（一）：产品定位与价值说明

## 1. DimOS 是什么

DimOS 本质上是一个面向机器人系统的本地运行时平台。

它的核心作用不是训练机器人，也不是单纯做某一种硬件驱动，而是把：

- 传感器接入
- 感知
- 导航
- 控制
- Agent
- 技能系统

装配成一个可以实际运行的机器人系统。

更准确地说，DimOS 是：

> 一套用 `Module + Blueprint + Runtime` 机制，把不同机器人能力模块组织成可运行机器人栈的平台。

它当前已经能覆盖：

- 机器狗
- 人形机器人
- 机械臂
- 无人机

## 2. 为什么要这样做

机器人系统天然不是一个单功能程序，而是多个能力协同工作的系统。

如果每换一种机器人、每上一个新项目、每增加一个新能力，都重新写一套系统，会带来几个明显问题：

- 重复开发多
- 硬件差异难管理
- 模块难复用
- 运行方式不统一
- 后续接入 Agent、技能、可视化、仿真成本高

因此需要一套统一运行框架，把“机器人能力开发”从一次性工程，变成可装配、可复用、可扩展的工程体系。

这就是 DimOS 的意义。

## 3. DimOS 的优势是什么

### 3.1 运行时复用

同一套 `Module + Blueprint + Worker` 机制，可以同时服务多种机器人，而不必为每个平台重写一套运行框架。

### 3.2 能力模块复用

感知、Agent、可视化、部分导航和技能模块，可以被不同机器人重复装配使用。

### 3.3 接口方式复用

不同机器人都按统一的 `Stream / RPC / Spec` 协作，而不是每个项目单独定义一套通信方式。

### 3.4 运行模式复用

同一条主逻辑可以切换：

- 真机
- 仿真
- 回放

而不是分别维护三套系统。

### 3.5 面向 Agent 扩展友好

DimOS 已经天然支持把机器人能力暴露为技能，再接给 Agent 或 MCP，这对后续智能化扩展非常有价值。

## 4. 为什么还要做云端设计

DimOS 目前的强项是“本地运行”，但如果系统规模扩大，会遇到新的管理问题：

- 不同机器人配置难统一管理
- 发布新版本时需要人工干预
- 模块版本、模型版本、配置版本难统一追踪
- 失败回滚缺乏标准流程
- 多台机器人缺乏统一发布和审计能力

所以，云端设计不是为了替代 DimOS 本地运行，而是为了补齐：

- 配置管理
- 发布管理
- 版本管理
- 回滚管理
- 审计管理

## 5. DimOS 云端化后的核心定位

DimOS 云端化后的正确定位不是：

- 让云端实时控制机器人

而是：

> 让云端负责管理，让机器人本地负责运行。

也就是：

- 云端负责配置、发布、审计、策略
- 机器人本地 Loader 负责拉取、校验、缓存、启动、健康检查、回滚
- DimOS Runtime 继续负责真实机器人系统运行

## 6. 一句话总结

DimOS 是一个面向通用机器人的本地运行时平台；做云端设计，是为了在不破坏本地运行稳定性的前提下，补齐配置、发布、回滚和审计能力，把 DimOS 从“可运行”提升为“可管理、可发布、可回滚”的机器人运行平台。
